最近 MiniMax 发了一篇技术报告，讲的是他们内部排查"模型无法说出马嘉祺"这个问题的过程。这个问题之前在社区里讨论挺多的，MiniMax 也把他们的排查思路和实验数据整理出来公开了。

我看完之后觉得这篇报告的含金量不低，它不只是在解释一个明星名字为什么说不出来，而是顺着这个问题把大模型训练里一个比较底层的缺陷给挖出来了。

## 实验一：分词器（Tokenizer）

### 分词器是怎么训练的

要搞清楚为什么说不出马嘉祺，得先从分词器说起。

大模型处理文字的方式和人不一样。它不是一个字一个字读进去的，而是先把文字切成一段一段的"词元"，也叫 token，然后以 token 为单位来处理。分词器就是负责这个切割工作的组件。

分词器本身也需要训练，但它的训练方式跟大模型完全不同，不需要大量 GPU，更多是统计层面的计算。目前主流的方法叫 BPE，Byte Pair Encoding，字节对编码。它的训练过程大概是这样：

第一步，把所有文字拆到最小单位，每个字都是一个独立的 token。第二步，在大规模语料库里做统计，看哪两个 token 挨在一起出现的次数最多。第三步，把频率最高的那对合并成一个新 token。第四步，不断重复这个过程，直到 token 总数达到预设上限。比如 MiniMax 的词表大约有 20 万个 token。

### "嘉祺"是怎么变成一个 token 的

现在问题就来了——"嘉祺"这两个字，在 MiniMax 的词表里是一个独立的 token，不是两个。

原因很简单：在 MiniMax 训练分词器的那个时间点，他们喂进去的互联网语料里包含大量关于时代少年团和马嘉祺的文本。算法扫描的时候发现，"嘉"字后面跟着"祺"的概率远高于跟其他字的概率，于是就把它们合并成了一个独立 token。

这个操作本身是合理的，目的是提高处理效率——同样一段文字，token 数越少，模型处理起来越快，上下文窗口也用得越省。

### 打包带来的陷阱

但这里有一个副作用，而且不太直观。

"嘉祺"被打包成一个独立 token 之后，它的命运就和"嘉"、"祺"这两个单字完全分开了。"嘉"和"祺"作为普通汉字，在各种文本里都会出现，模型训练的时候会反复练到它们。但"嘉祺"这个组合 token，只会在提到马嘉祺的时候出现。

问题就出在后训练阶段——也就是 SFT，监督微调。这个阶段用的是精选的对话数据，用来教模型怎么聊天、怎么遵循指令。这批数据里，提到"马嘉祺"的样本不到 5 条。这意味着"嘉祺"这个 token 在长达数周的后训练过程中，几乎一次都没有被练到。

与此同时，那些在后训练数据里大量出现的 token，比如 tool_call 标记、代码符号等，会持续更新自己在向量空间里的位置，顺带着把"嘉祺"这种没练到的 token 往旁边挤。挤到最后，"嘉祺"跑偏了——模型还认识它，但在需要输出它的时候，选了"佳琪"、"琪琪"这些发音相近、但实际上练过的 token。

如果当初"嘉祺"没有被合并，保持"嘉"和"祺"两个独立单字，反而不会有这个问题，因为这两个字会跟着其他词汇一起被反复练到，永远不会出现遗忘。这是一个有点反直觉的地方——分词器越"聪明"，有时候反而越容易埋坑。

### 为什么其他模型可能没这个问题

不同厂商的分词器训练语料不一样，合并策略也不一样。

如果某个模型的分词器训练语料里娱乐内容比例低，或者合并阈值设得比较保守，"嘉祺"可能就不会被合并，依然是两个独立单字。这样在后训练阶段，即使没有专门练到"马嘉祺"，模型也不会出现说不出来的情况。

所以这个 bug 不是所有模型都有，但背后的机制是通用的——任何因为在后训练数据里出现频次极低而没有被充分训练的独立 token，都可能发生类似的退化。

### 跨模型对比：不同分词器的切割方式

为了更直观地说明这个问题，我对几个主流模型的分词器做了对比，看看它们对同一段文字的处理方式有什么不同。

拿"马嘉祺"来测，各家分词器的切法差异很明显。

【插入 0301_马嘉祺分词展示.png】

GPT-2 对中文基本没有优化，切出 9 个 token，每个汉字都拆碎了。Llama 切成 5 个。Qwen 和 DeepSeek 都切成 3 个——马、嘉、祺，三个字各自独立。MiniMax 切成 2 个——"马"一个，"嘉祺"打包成一个。

有意思的是"迷失传奇"这个词。

【插入 0302_迷失传奇分词展示.png】

Qwen 直接把四个字整体打包成 1 个 token，压缩率最高。DeepSeek 切成 2 个——迷失、传奇。MiniMax 同样是 2 个——迷失、传奇。反而是 GPT-2 和 Mistral 切得最碎，8 个 token。

所以不同模型的分词策略差异很大，打包的词也不一样。Qwen 把"迷失传奇"整体打包，MiniMax 把"嘉祺"整体打包——打包的选择，取决于各家训练分词器时喂的语料里哪些词组合出现的频率足够高。打包本身是为了提升效率，但如果这个打包后的 token 在后训练数据里几乎没出现过，问题就从这里开始了。

### 实验演示：分词器的选词逻辑

我这里做了一个简单的实验来展示这个机制。

我用两种方式去问模型。第一种是引导式：请把"马"、"嘉"、"祺"这三个字连起来输出，不要说别的。第二种是直接式：重复一下马嘉祺三个字，不要说别的。

【插入 0401_展示选词逻辑.png】

看图里的结果。引导式问法里，输入被分词器切成了独立的"马"、"嘉"、"祺"三个单字，模型回复是"马嘉祺"，正确。直接式问法里，"嘉祺"作为组合 token 整体进入模型，回复变成了"马璟祺"，出错了。

下半部分也可以看一下，同样是"马嘉祺"这个词，在不同的上下文里分词结果不一样。单独出现时切成"马"+"嘉祺"；但在逐字引导的语境里，"嘉"和"祺"会被拆开成独立单字。这说明分词不是固定的，上下文会影响切法，而切法直接决定了模型能不能正常输出这个词。

问题出在哪个环节就很清楚了：不是模型不认识马嘉祺，而是一旦"嘉祺"作为整体 token 进入生成流程，退化就发生了。

### 小结

分词器在这里扮演了触发问题的角色。它把"嘉祺"合并成一个独立 token，使得这个 token 的命运完全依赖于后训练数据里有没有出现过"马嘉祺"。而现实是后训练数据里几乎没有。这是整个问题链条的起点。

## 实验二：Base 模型 vs Chat 模型

### 进入实验二

但分词器不是唯一的原因。MiniMax 的排查结论是——这是分词器和 SFT 共同造成的问题。分词器决定了"嘉祺"的存在形式，SFT 决定了它的最终命运。

为了把这个逻辑说清楚，他们做了一组对比实验，对象是 Base 模型（预训练完、还没有经过 SFT 的版本）和 Chat 模型（SFT 之后的版本）。

我把这四个实验在这里展示一下，同时对 MiniMax 报告里的数据做一个解析。

### **实验 1 & 2：Embedding Norm + LM Head Norm 检查**

前两个实验放在一起看。

第一个实验检查的是"嘉祺"在 Base 模型输入层（embedding 层）的向量模长是否正常。如果一个 token 在预训练阶段根本没练到，它的向量通常会非常小，接近随机初始化的状态，在分布图里会明显偏离主体，落到左侧边缘去。MiniMax 检查的结果是，"嘉祺"的向量模长落在正常分布范围内，没有异常。

第二个实验检查的是输出层（lm_head）——也就是模型在生成文字时最后一步的参数。结果同样正常。这说明在 SFT 之前，模型不只是认识"嘉祺"，也能正常地把它生成出来。

【插入 0101_minimax-pretrain.png】

图里横轴是模长的数值范围，纵轴是对应数值的 token 数量，红色虚线标注的是"嘉祺"对应的 token 190467。左边是输入层，右边是输出层，两张图里它都落在分布的主体范围内，没有任何异常。

结论：Base 模型的输入端和输出端都没有问题，预训练阶段"嘉祺"被充分训练了。问题是在 Base 到 Chat 这个过程中引入的。

### **实验 3 & 4：LM Head 变化幅度 + 全词表退化扫描**

前两个实验确认了 Base 模型是好的，那问题就锁定在 SFT 阶段。后两个实验是量化地看 SFT 到底造成了多大的损伤。

第三个实验的方法是计算 SFT 前后，"嘉祺"对应的 lm_head 向量的余弦相似度。余弦相似度衡量的是两个向量的方向有多接近，值越接近 1 说明方向基本没变，越低说明偏得越厉害。

第四个实验是把这套检查方法扩展到整个词表，对全部约 20 万个 token 都过一遍，看看"嘉祺"的情况是不是个例。方法是对每个 token 在 SFT 前后的 lm_head 参数做 L2 范数的作差，差值越大说明变化越剧烈，退化越严重。

【插入 0102_minimax-sft.jpg】

左边这张是余弦相似度分布，横轴是相似度数值，纵轴是 token 数量。大部分 token 都集中在 0.95 到 1.0 之间，而 token 190467 的值是 0.9568，落在了整体分布的左侧边缘，明显靠左。同时 Norm 变化也很大。这说明 SFT 过程把"嘉祺"的输出向量推到了一个不对的方向上。

右边是 lm_head L2 Diff，直接看 SFT 前后参数的变化量，值越大说明动得越多。token 190467 的红线跑到了分布右侧的长尾区域，偏移量远高于大多数 token。

两张图合在一起说明一件事：问题出在输出层，SFT 把"嘉祺"的生成能力带歪了。

把扫描扩展到全词表之后，结果是约 4.9% 的 token 发生了显著退化。退化的 token 里，排在前面的有"传奇私服"、"无痛人流"这类 SEO 关键词——它们在预训练的互联网数据里大量出现，被分词器打包成了独立 token，但在后训练的对话数据里完全没有出现过，退化机制和"嘉祺"一模一样。

占比最大的一类是日语口语和博客模板类 token，接近 40%。这个发现引出了另一个问题，后面会讲到。

结论：**这不是"嘉祺"一个 token 的问题，是系统性的。凡是后训练数据里没练到的 token，都在朝不对的方向漂移。**

顺便说一个我在测试过程中发现的有意思的事。千问说不出"迷失传奇"、"新开传奇"、"法战组合"，我手边在用的小米 MiMo 2.5 也说不出"法战组合"，MiniMax 自己也在报告里点名了"传奇私服"。

传奇这个游戏，当年的广告真是无孔不入。没想到这些词在大模型里留下了这么一个集体失忆的痕迹。

## 实验三：微调实验（LoRA）

进入实验三。既然问题出在 lm_head 层，我来试一下针对这个层做定向微调，看能不能把退化的 token 拉回来。

我用 LLaMA Factory 对千问 2.5 7B 做了 LoRA 微调，跑了两次。

第一次调了 Q projection、V projection 和 lm_head，LoRA 秩 8，缩放系数 16，学习率 5e-5，训练 10 轮。

【插入 0901_lora1csv.png】

用同样的方法重新扫一遍，"迷失传奇"已经不在前十了。排名靠前的换成了狐月山、新开传奇、法战组合这些词，余弦相似度也比微调前略有回升。说明第一次微调对"迷失传奇"有一定效果，但整体改善有限。

第二次加大力度，调Q projection、V projection、lm_head 和 embed_tokens，学习率调到 1e-4，LoRA 秩 16，缩放系数 32，训练 50 轮。先用同样的方法重新扫一遍。

第一个是微调前，第二个是微调后，重点看右下角的 lm_head L2 Diff，也就是输出层向量相对于 Base 模型的偏移量分布。

微调前，整个词表的分布峰值在 0.02 附近，而"迷失传奇"的偏移值是 0.0649，远远跑到了分布曲线的右侧边缘，跟大部分 token 完全不在一个区间里。这是它说不出口的直接原因——输出层的向量偏离 Base 模型太远，生成概率被压得很低。

微调后，"迷失传奇"的绝对偏移值变化不大，从 0.0649 降到 0.0634。但关键是整体分布的峰值从 0.02 移到了 0.04 到 0.05 之间——整个词表都动了，"迷失传奇"从原来的极值离群点，落回到了正常变动范围内。它现在不再是一个异类，而是落在了模型输出层正常变动的范围里。

左下的余弦相似度也可以看一下，从 0.9846 回升到 0.9857，涨了 0.0011。数字很小，但方向是对的，说明输出向量在往 Base 模型的原始方向回偏。在 LoRA 秩 16 的限制下，同时作用于多个模块，能在方向上往回扭一点，说明 50 轮的训练确实在纠偏。

不过 50 轮也带来了副作用——一提"传奇"两个字，模型就开始循环输出"迷失传奇"，停不下来。过拟合了，训练数据太单一、轮次太多。这也印证了 MiniMax 报告里说的，修复低频 token 退化，数据多样性和训练比例的控制同样重要。

当然这个实验不是为了做一个完整的修复方案——LoRA 微调 lm_head 本身就是一个比较受限的操作，覆盖范围也有限。它的意义在于验证：MiniMax 报告里说的"后训练阶段可以补救分词器带来的问题"这个思路，在实际操作中是成立的。问题是可以被干预的，方向是对的。

## 实验意义

### "说不出马嘉祺"和"日语蹦出俄语"是同一个问题

我在这里稍微展开说一下这件事的工程意义，因为我觉得 MiniMax 这篇报告最有价值的地方不是解决了马嘉祺的问题，而是把两个看起来完全不相关的问题统一到了同一个框架里。

第一个问题是说不出马嘉祺，第二个问题是模型在处理日语的时候偶尔会混入俄语或韩语字符。这个语言混杂的问题之前一直没找到确切原因。

MiniMax 的全词表扫描给出了答案：29.7% 的日语 token 在 SFT 阶段发生了显著退化，远高于其他语种（中文 3.9%，英文 3.5%，俄语 3.7%，韩语 3.3%）。原因是后训练数据里日语内容覆盖严重不足，日语 token 没有被充分练到，于是在向量空间里发生了漂移。而漂移的方向，恰好挨着俄语和韩语的 token 区域，所以在输出的时候会走错。

这两个现象——"嘉祺"被替换成"佳琪"，以及日语里蹦出俄语——背后是同一套机制：某个 token 在 SFT 阶段没被练到，它的 lm_head 向量被周围高频 token 推着跑偏，最终在生成的时候找错了邻居。

### 三个工程层面的价值

**第一个，从语义评估到 token 监控。** 以前做 SFT 评估，主要看模型有没有变聪明——数学对不对、代码好不好。MiniMax 的这套实验说明，这还不够。词表健康度也需要纳入监控，需要定期检查全词表里有没有 token 在悄悄退化。这是一个监控维度的补充。

**第二个，揭示多语言混杂的底层原因。** 以前遇到语言混杂问题，一般会去查语料有没有污染，数据有没有混入。这次的发现说明，即使数据是干净的，如果某个语种在 SFT 阶段覆盖不够，一样会出现混杂。病根不是数据脏，是练得少。找到了真正的原因，修复方案就变得便宜了很多——不需要重新洗数据，定向补充覆盖不够的 token 就可以。

**第三个，证明后训练可以补偿分词器的先天问题。** 分词器是在预训练开始之前就定好的，模型一旦开始训练就没法改词表了。以前的认知是，分词器设计不好，那就是先天不足，没有太好的办法。MiniMax 证明了，通过 SFT 阶段的定向数据补充，可以把跑偏的 token 拉回来。分词器的某些"错误决策"，在后训练阶段有机会纠正。

### 解决方案

MiniMax 报告里提到了三种修复思路。第一种是在 SFT 数据里混入一定比例的预训练语料，利用预训练数据天然覆盖面广的特点来缓解稀疏 token 的退化。第二种是针对覆盖不够的 token 定向合成训练样本，比他们用的"全词表复读"方案更精准，但需要维护一套 token 覆盖度的监控体系。第三种是词表裁剪加继续预训练，把那些在目标场景下永远用不到的 token 直接从词表里移除，然后重新对齐 embedding 空间。这个方案改动最大，但能从根本上解决问题。

三种方案各有代价，MiniMax 目前验证有效的是第一种的变体——构造覆盖全词表的合成数据，保证每个 token 都被练到一个下限频率。效果是：日语混入俄语的比例从 47% 降到了 1%，全词表的 lm_head 余弦相似度从最低 0.329 提升到全部高于 0.97。

## 结尾

好，今天三个实验讲完了。我做这几个实验的目的，是想把 MiniMax 报告里的结论在自己这里验证一遍，看看这套机制是不是真实存在的，是不是可以复现的。

结论是可以的。在千问模型上，同样的问题用同样的方法可以观察到，LoRA 微调 lm_head 的修复思路也有效果。

MiniMax 报告的价值不只是解释了马嘉祺的事，而是提供了一套可以落地的诊断方法和修复方向。对于任何在做大模型后训练的团队来说，token 覆盖度的监控这件事，之后可能会成为一个常规检查项。
